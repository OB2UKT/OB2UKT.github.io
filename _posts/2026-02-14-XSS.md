## Khái niệm về XSS
XSS hay Cross-site script là một lỗ hổng bảo mật đến từ phía client-site, lỗ hổng này bắt nguồn từ HTML injection khi mà attacker hoàn toàn có thể bỏ script vào trang web và thực thi js, từ đó thực hiện tấn công đánh cắp thông tin người dùng thậm chí khai thác sâu vào hệ thống có khả năng gây ra RCE.

## Stored DOM XSS
Ngay phần giới thiệu lab ta đã biết lỗ hổng này sẽ đến từ phần comment trong 1 blog, chính vì thế ta sẽ tập trung vào phân tích để tiết kiệm thời gian. 

![Intercepted Request](assets/img/material_posts/XSS/img28.png){: width="800" height="500" }

Vậy thì ở đây ta có thể thấy được trang web thực hiện xử lý việc load các comment trong 1 bài post thông qua 1 file "loadCommentsWithVulnerableEscapeHtml.js" và ta sẽ phân tích chi tiết file này. 

```javascript
function loadComments(postCommentPath) {
    let xhr = new XMLHttpRequest();
    xhr.onreadystatechange = function() {
        if (this.readyState == 4 && this.status == 200) {
            let comments = JSON.parse(this.responseText);
            displayComments(comments);
        }
    };
    xhr.open("GET", postCommentPath + window.location.search);
    xhr.send();

    function escapeHTML(html) {
        return html.replace('<', '&lt;').replace('>', '&gt;');
    }

    function displayComments(comments) {
        let userComments = document.getElementById("user-comments");

        for (let i = 0; i < comments.length; ++i)
        {
            comment = comments[i];
            let commentSection = document.createElement("section");
            commentSection.setAttribute("class", "comment");

            let firstPElement = document.createElement("p");

            let avatarImgElement = document.createElement("img");
            avatarImgElement.setAttribute("class", "avatar");
            avatarImgElement.setAttribute("src", comment.avatar ? escapeHTML(comment.avatar) : "/resources/images/avatarDefault.svg");

            if (comment.author) {
                if (comment.website) {
                    let websiteElement = document.createElement("a");
                    websiteElement.setAttribute("id", "author");
                    websiteElement.setAttribute("href", comment.website);
                    firstPElement.appendChild(websiteElement)
                }

                let newInnerHtml = firstPElement.innerHTML + escapeHTML(comment.author)
                firstPElement.innerHTML = newInnerHtml
            }

            if (comment.date) {
                let dateObj = new Date(comment.date)
                let month = '' + (dateObj.getMonth() + 1);
                let day = '' + dateObj.getDate();
                let year = dateObj.getFullYear();

                if (month.length < 2)
                    month = '0' + month;
                if (day.length < 2)
                    day = '0' + day;

                dateStr = [day, month, year].join('-');

                let newInnerHtml = firstPElement.innerHTML + " | " + dateStr
                firstPElement.innerHTML = newInnerHtml
            }

            firstPElement.appendChild(avatarImgElement);

            commentSection.appendChild(firstPElement);

            if (comment.body) {
                let commentBodyPElement = document.createElement("p");
                commentBodyPElement.innerHTML = escapeHTML(comment.body);

                commentSection.appendChild(commentBodyPElement);
            }
            commentSection.appendChild(document.createElement("p"));

            userComments.appendChild(commentSection);
        }
    }
};
```
Có thể thấy hàm loadComments là hàm mẹ chịu trách nhiệm xử lý trong việc lấy comment và xử lý nó trên bài post. Vậy thì ở đây ta thấy được hàm này sẽ thực hiện lấy các comment thông qua "xhr.open("GET", postCommentPath + window.location.search);" 
Sau đó ó sẽ thực hiện xử lý các comment này như 1 object cụ thể. Ở đây đáng chú ý là ta thấy hàm escapeHTML được dùng để xử lý các giá trị untrusted do người dùng nhập vào như comment, name.

Tuy nhiên ở đây hàm xử lý dấu lại gặp vấn đề khi sử dụng hàm replace() để chống lại XSS, bởi vì replace chỉ có thể thay đổi ký tự lần đầu gặp, chính vì thế nếu như người dùng nhập chuỗi <>/Chuỗi khai thác/ thì hoàn toàn có thể bypass được cơ chế của replace().

## Khắc phục XSS
Như đã kể ở trên ta đã lần lượt tìm hiểu qua 3 loại XSS phổ biến bao gồm: Reflected XSS, Stored XSS, DOM-based XSS. Để có thể khắc phục được lỗ hổng XSS ta có 1 số biện pháp và cấu hình như: Sử dụng kết hợp Output Encoding + CSP (Content Security Policy). Vậy thì giờ ta sẽ tìm hiểu sau về 2 cơ chế này. 

### Output Encoding 
Đầu tiên đến với Output Encoding hiểu ngắn gọn thì đây là 1 cấu hình mà ở đó nó cho phép dev thực hiện mã hóa dữ liệu hiển thị trên trang web. Vậy thì đối với lỗ hổng XSS nó sẽ giúp mã hóa toàn bộ những payload mà attacker đã thực hiện tấn công. Giả sử attacker có chèn payload để thực hiện Reflected XSS. 

```html
<!-- SAMPLE -->
<img src="||PAYLOAD HERE||"> 

<!-- PAYLOAD -->
<img src="" onload="alert(document.domain)"> 

<!-- Endcode Output Config -->
<img src="&quot;onerror=&quot;alert(document.domain)"&gt;
```
Có thể thấy được sự trực quan của cấu hình này khi mà nó có thể thực hiện mã hóa trước khi xử lý và in ra, điều này giảm đáng kể lỗ hổng XSS khi mà nó hoàn toàn không thể thực thi được payload. Tuy nhiên việc cấu hình HTML Output Encoding còn phụ thuộc rất nhiều vào việc dev có cấu hình chuẩn hay không và nó phải có tính linh hoạt trong 1 số trường hợp cụ thể, chính vì thế sai sót là điều không thể tránh khỏi. Ta lại đến với lớp bảo mật thứ 2 đó chính là CSP (Content Security Policy). 

### Content Security Policy 
Đây chính là config giúp là chốt chặn cuối cùng và là mảnh ghép phù hợp đối với **Output Encoding** kể trên, nếu như ta gặp vấn đề về việc cấu hình **HTML Output Encoding** gây trót lọt payload thì attacker có thể khai thác XSS. Tuy nhiên với chính sách của CSP sẽ ngăn theo dạng white-list, yêu cầu chỉ thực thi 1 số script đã được quy định trước đó, tránh tải file không rõ, truy cập từ phía bên ngoài các địa chỉ không nằm trong danh sách an toàn, hoặc hệ sinh thái của nó. Điều này giúp ngăn chặn đáng kể khả năng khai thác của XSS (ví dụ như thực thi payload độc hại <script>...</script>). 

> Có thể thấy được sự kết hợp giữa Content Security Policy và Output Encoding là hợp lý và có thể dùng để ngăn chặn XSS. Tuy nhiên ta nên kết hợp cả 2 cấu hình này để đạt hiệu quả tuyệt đối, tránh việc tin tưởng vào HTML Output Encoding hoặc ngược lại.
